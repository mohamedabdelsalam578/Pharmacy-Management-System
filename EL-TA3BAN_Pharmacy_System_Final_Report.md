## 3. System Architecture

### 3.1 Architectural Overview

The EL-TA3BAN Pharmacy Management System implements a modern service-oriented architecture with clear separation of concerns and layers. This design approach ensures that the system is maintainable, extensible, and follows established software engineering principles.

Our architecture consists of four distinct layers:

1. **‚úì Model Layer**: Core data entities that represent the business domain
   - User hierarchy (Admin, Patient, Doctor, Pharmacist)
   - Healthcare entities (Medicine, Prescription, Order, etc.)
   - Financial components (Wallet, Transaction)

2. **‚úì Service Layer**: Business logic implementation
   - PharmacyService: Central workflow coordination
   - AuthenticationService: Security and user management
   - Role-specific services (AdminService, DoctorService, etc.)

3. **‚úì Utility Layer**: Helper functions and tools
   - FileHandler: Data persistence operations
   - ConsoleUI: User interface components
   - SystemTools: Common utilities and validations

4. **‚úì Persistence Layer**: Data storage and retrieval
   - Text file storage for basic entities
   - Serialization for complex objects like wallets
   - In-memory caching for performance

### 3.3 Key Relationships and Associations

#### 3.3.1 Primary Class Relationships

| Class Relationship | Type | Cardinality | Description |
|-------------------|------|-------------|-------------|
| User ‚Üí Roles | Inheritance | 1 : 4 | One abstract User class inherited by all role classes |
| Patient ‚Üî Prescription | Association | 1 : * | Patient receives multiple prescriptions |
| Doctor ‚Üî Prescription | Association | 1 : * | Doctor creates multiple prescriptions |
| Pharmacist ‚Üî Pharmacy | Association | * : 1 | Multiple pharmacists staff one pharmacy |
| Prescription ‚Üî Medicine | Association | * : * | Many prescriptions contain many medicines |
| Patient ‚Üî Wallet | Composition | 1 : 1 | Each patient has one dedicated wallet |
| Wallet ‚Üî Transaction | Composition | 1 : * | Wallet contains multiple transaction records |
| Patient ‚Üî Consultation | Association | 1 : * | Patient can participate in multiple consultations |
| Doctor ‚Üî Consultation | Association | 1 : * | Doctor can conduct multiple consultations |

#### 3.3.2 Detailed Relationship Types

| Relationship | Type | Description |
| --- | --- | --- |
| **‚úì User ‚Üí Role Classes** | Inheritance | "is-a" relationship with shared base functionality |
| **‚úì Patient ‚Üí Wallet** | Composition | Strong ownership where wallet cannot exist without patient |
| **‚úì Wallet ‚Üí Transaction** | Composition | Transactions belong to and exist only within a wallet context |
| **‚úì Wallet ‚Üí Card** | Composition | Cards are owned by and managed within a wallet |
| **‚úì Patient ‚Üî Order** | Association | Bidirectional relationship without lifecycle dependency |
| **‚úì Doctor ‚Üî Prescription** | Association | Creation relationship without strict ownership |
| **‚úì Prescription ‚Üî Medicine** | Many-to-Many | Medicines can be in multiple prescriptions with quantities |
| **‚úì Consultation ‚Üî Message** | Composition | Messages belong to and cannot exist outside consultations |
| **‚úì All Models ‚Üí Serializable** | Implementation | Interface implementation for data persistence |
